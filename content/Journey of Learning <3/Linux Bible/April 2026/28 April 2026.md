**1. Ringkasan Konsep Kunci**

- **Proses dan PID:** Setiap program yang berjalan di sistem Linux disebut proses. Kernel Linux memberikan nomor unik bernama _Process ID_ (PID) untuk melacak setiap proses tersebut secara spesifik.
- **Snapshot vs Real-time:** Perintah `ps` hanya mengambil "foto" (_snapshot_) dari status proses pada detik perintah tersebut dieksekusi, sedangkan `top` menampilkan aktivitas CPU dan memori secara dinamis dan diperbarui terus-menerus (biasanya tiap 3 detik).
- **Foreground vs Background:** Proses di _foreground_ akan mengunci terminal sampai proses itu selesai. Jika Anda menambahkan tanda ampersand (`&`) di akhir perintah, proses akan berjalan di _background_, sehingga Anda bisa lanjut mengetik perintah lain di terminal yang sama.
- **Sinyal (Signals) sebagai Alat Kontrol:** Perintah `kill` sebenarnya tidak langsung "membunuh", melainkan mengirimkan sinyal ke suatu proses. Sinyal default adalah **SIGTERM (15)** yang meminta proses berhenti dengan aman, sedangkan **SIGKILL (9)** memaksa proses berhenti seketika tanpa ampun.

**2. Perintah / Syntax Wajib Hafal**

- **`ps aux`**: Menampilkan daftar semua proses dari semua _user_ dengan informasi lengkap (termasuk %CPU dan %MEM).
- **`ps aux | grep <nama_aplikasi>`**: Cara tercepat untuk mencari PID dari aplikasi yang namanya Anda ketahui.
- **`top`**: Membuka _dashboard_ pemantauan proses. (Gunakan tombol **`q`** untuk keluar dari tampilan ini).
- **`<perintah> &`**: Contoh: `sleep 100 &`. Memulai sebuah program agar langsung berjalan di _background_.
- **`jobs`**: Melihat daftar proses _background_ yang berjalan di sesi terminal Anda saat ini.
- **`kill <PID>`**: Mengirim sinyal berhenti aman ke nomor PID tertentu.
- **`kill -9 <PID>`**: Memaksa proses dengan PID tersebut untuk langsung mati.

**3. Latihan Praktis (45 Menit di Distrobox)** Buka terminal Anda dan lakukan skenario berikut secara berurutan:

- **Langkah 1 (Simulasi Proses Background):** Buat sebuah proses bohongan yang memakan waktu lama di _background_ dengan mengetikkan: `sleep 3000 &`. Terminal akan merespons dengan menampilkan _Job ID_ dan _PID_, contohnya `12345`.
- **Langkah 2 (Monitoring):** Ketik **`top`** dan tekan Enter. Perhatikan kolom `%CPU` dan `%MEM` untuk melihat proses apa yang memonopoli resource. Temukan di mana posisi proses `sleep` Anda, lalu tekan **`q`** untuk kembali ke _prompt_ utama.
- **Langkah 3 (Investigasi & Filter):** Karena `top` terkadang terlalu cepat berubah, gunakan perintah statis. Ketik **`ps aux | grep sleep`**. Temukan nomor PID dari perintah `sleep 3000` yang Anda jalankan di Langkah 1.
- **Langkah 4 (Eksekusi Penghentian):** Hentikan proses tersebut secara aman dengan mengetik **`kill <PID_yang_ditemukan>`**.
- **Langkah 5 (Verifikasi):** Ketik **`jobs`** atau tekan Enter sekali lagi. Anda akan melihat pesan seperti `+ Terminated sleep 3000`, yang menandakan proses sudah mati.

**4. Cara Tahu Bahwa Anda Sudah Menguasai Sesi Ini** Anda dianggap telah berhasil dan menguasai Sesi Mod 1 jika Anda bisa melakukan tiga hal ini tanpa melihat sontekan:

1. Bisa menyebutkan perintah untuk melihat aplikasi apa saja yang sedang mengonsumsi CPU tertinggi di server secara _real-time_.
2. Bisa menyaring (_filter_) output proses yang ratusan baris hanya untuk menemukan nomor PID dari _service_ yang spesifik (misalnya, mencari proses `nginx` atau `mysql`).
3. Memahami alasan mengapa seorang sysadmin sebaiknya mencoba menggunakan perintah `kill <PID>` terlebih dahulu sebelum menggunakan `kill -9 <PID>` saat mengatasi aplikasi yang macet.

# Fungsi Grep dalam Linux
Perintah `grep` akan menjadi salah satu senjata utama Anda kelak sebagai seorang DevOps untuk melakukan investigasi dan _troubleshooting_. Pada dasarnya, `grep` bertugas mencari pola teks spesifik di dalam sebuah file atau input standar yang diberikan kepadanya.

Berikut adalah panduan praktis dan efisien untuk menggunakan `grep` saat Anda mencari error di dalam file log:

**1. Pencarian Error Dasar** Format default untuk pencarian adalah `grep [opsi] pola [file]`. Jika Anda ingin mencari kata "error" di dalam sebuah file log spesifik, Anda cukup mengetikkan: `grep "error" /var/log/syslog`

**2. Mengabaikan Huruf Besar/Kecil (Case Insensitive)** Sistem sering kali menuliskan log dengan huruf yang bervariasi, seperti "Error", "ERROR", atau "error". Agar `grep` mengabaikan perbedaan huruf besar dan kecil, gunakan parameter **`-i`**. `grep -i "error" /var/log/syslog`

**3. Melihat Kronologi Kejadian (Context Control)** Hanya melihat baris error saja biasanya tidak cukup. Seorang DevOps harus tahu _apa yang memicu_ error tersebut. Anda bisa menampilkan beberapa baris teks sebelum atau sesudah pesan error ditemukan:

- **`-B NUM` (Before):** Menampilkan sekian baris _sebelum_ teks yang cocok.
- **`-A NUM` (After):** Menampilkan sekian baris _sesudah_ teks yang cocok.
- **`-C NUM` (Context):** Menampilkan sekian baris _sebelum dan sesudah_ teks yang cocok.

Contoh untuk melihat kronologi 3 baris sebelum dan sesudah error terjadi: `grep -C 3 -i "error" /var/log/syslog`

**4. Menemukan Nomor Baris** Untuk memudahkan Anda melacak lokasi error jika nantinya file tersebut harus diedit (seperti menggunakan `vim`), tambahkan parameter **`-n`** yang berfungsi mencetak nomor baris pada setiap hasil yang cocok. `grep -n -i "error" /var/log/syslog`

**5. Mencari ke Seluruh File (Recursive Search)** Di dunia nyata, direktori `/var/log/` berisi puluhan file. Jika Anda tidak tahu persis file log mana yang mencatat error dari aplikasi Anda, gunakan parameter **`-r`** atau **`-R`** agar `grep` menelusuri setiap file di dalam folder dan subfolder secara rekursif. `grep -R -i "error" /var/log/`

**6. Menyaring Pesan yang Tidak Penting (Invert Match)** Sering kali kata "error" memunculkan banyak sekali baris yang mungkin sebagian besar adalah _error_ yang tidak berbahaya. Anda bisa membuang atau mengecualikan baris yang mengandung kata spesifik menggunakan parameter **`-v`**. Contoh ini akan mencari "error" tetapi _mengabaikan_ baris yang mengandung kata "timeout": `grep -i "error" /var/log/syslog | grep -v "timeout"`

**7. Menghitung Jumlah Error** Jika Anda sedang menganalisis seberapa parah masalah yang terjadi, Anda bisa menghitung seberapa sering error tersebut muncul. Parameter **`-c`** akan berhenti mencetak teks barisnya dan hanya mengeluarkan angka jumlah _error_ yang ditemukan. `grep -c -i "error" /var/log/syslog`

Sebagai latihan di Distrobox Anda hari ini, cobalah jalankan perintah `grep -n -i "error" /var/log/syslog` (atau `/var/log/messages`, tergantung pada distribusi Linux yang Anda gunakan) dan perhatikan bagaimana terminal dengan cepat menyaring data tersebut.
